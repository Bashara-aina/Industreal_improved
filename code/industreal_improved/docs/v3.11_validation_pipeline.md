# v3.11 Validation Pipeline — How We Stop the Cycle

**Date:** 2026-07-22

## The problem we're solving

We kept launching training runs that failed in the same ways:
- mAP stuck at 0.05
- Activity head never trains
- Pose MAE never drops
- PSR F1 stuck at 0.17

Each run took hours, then we'd discover the issue and patch it, then launch another run. The cycle repeated.

## The fix: Pre-flight validation BEFORE training

**`scripts/verify/preflight_v310.py`** runs 53 checks across 10 categories BEFORE we launch a training run. If ANY check fails, we don't launch.

## Bugs we caught with pre-flight

### 1. LR imbalance (Cat 5: OPT)
- **Before**: Activity/pose/PSR heads used base_lr=2e-5, detection used 1000x multiplier
- **After**: Activity/pose/PSR heads use 50x multiplier (1e-3), detection still 1000x
- **Why this matters**: Non-det heads were starved 1000x; couldn't learn

### 2. Bias buffer not persistent (Cat 7: BIAS)
- **Before**: `running_pos_ratio` was plain Python float, reset on reload
- **After**: Registered as persistent buffer, persists across save/load

### 3. update_logit_bias overwrote per-class gradients
- **Probe**: 500-step overfit-200 showed -16.5% BG conf, +15.3% FG/BG separation
- **Fix**: Default OFF, requires `--use-logit-bias-update` flag

## Why this won't fail like the others

Before launching, we verified:
1. ✓ All 4 heads get gradients
2. ✓ Detection overfits in 100 steps (loss 2.64 → 0.04)
3. ✓ All optimizer LRs are correct
4. ✓ All loss functions compute correctly
5. ✓ All data pipeline checks pass
6. ✓ YOLOv8 baseline (0.59 mAP) confirms data supports high mAP

If v3.11 still doesn't improve, the next check is the CONV probe's finding:
the model CAN learn — but maybe multi-task training interferes. We'd then need to:
1. Train each head separately and verify each can reach SOTA
2. Add per-head gradient isolation
3. Try curriculum learning (warm up one head at a time)

