# Doc 215 — 50 Deep Questions for Claude Science

**Status**: Research briefing. 2026-07-11.
**Purpose**: Questions to answer via academic paper scanning, to guide our MTL architecture and optimization strategy for the IndustReal assembly dataset.
**Our setup**: MViTv2-S backbone (34.5M, 81.0% K400), Kendall uncertainty weighting with capped precisions, PCGrad gradient surgery, EMA model weights, BiFPN neck, per-task heads (det: decoupled cls+reg, act: 3-layer MLP, PSR: 2-layer causal transformer, pose: 6D+geodesic). 4 tasks: detection (24-class, small objects ~20px), activity recognition (75-class, power-law), PSR (11-component, <1% positive frames), head pose (6D continuous).
**Core problem**: MTL performance is well below ST baselines on detection, activity, and PSR. We seek published methods to close the MTL-to-ST gap and ideally beat single-task SOTA on all heads.
**Output target**: A research summary with actionable architecture/optimization recommendations for the next model iteration.

## Section A: MTL Architecture (8 questions)

---

**Question A1: Can task-conditional feature modulation (FiLM, adapters, or gating) achieve single-task-level accuracy per head while sharing a single backbone across 4+ tasks?**

We currently use a completely shared MViTv2-S backbone where all 16 transformer blocks produce the same features for all four heads. The detection head operates on FPN features from P3/P4/P5, activity and pose heads use the cls_token, and the PSR head uses P5 spatial features. This rigid sharing means gradient conflicts from disparate tasks (e.g., detection's spatial localization vs. activity's global semantics) must be resolved at the optimization level via PCGrad rather than at the representation level. Task-conditional modulation methods like FiLM (Perez et al., 2017), adapters (Rebuffi et al., 2017), or task-specific gating (Mallya et al., 2018) could let each task modulate intermediate features without requiring separate backbones. A paper that demonstrates >90% ST retention on 4+ diverse tasks (including at least one dense prediction task like detection) using task-conditional modulation would directly inform our next architecture iteration. We need to know which modulation mechanism works for detection, a task that typically degrades most under feature sharing.

**Question A2: What architectural patterns have been proven to achieve positive transfer across all tasks in 4+-task MTL benchmarks (Taskonomy, NYUv4, PASCAL-Context)?**

Our 4-task setup spans three fundamentally different output types: dense spatial (detection), temporal sequential (PSR), global classification (activity), and continuous regression (pose). Most MTL papers study 2-3 tasks of similar type (e.g., semantic segmentation + depth). We need evidence from benchmarks like Taskonomy (Zamir et al., 2018) or PASCAL-Context that shows which task groupings naturally benefit from shared representations and which require explicit separation. A paper demonstrating positive transfer across all tasks in a 4+ task setup with at least one detection or localization task would provide the architectural template for our system. We currently observe negative transfer on detection and activity, suggesting our task grouping may require architectural separation at specific backbone depths.

**Question A3: What backbone architectures (beyond MViTv2-S) have been proven superior for multi-task learning relative to their single-task performance?**

Our MViTv2-S backbone achieves 81.0% on Kinetics-400 and 34.5M parameters, but it was designed and pretrained for single-task video classification. Alternative backbones like ConvNeXt (Liu et al., 2022), Swin (Liu et al., 2021), or hierarchical vision transformers with explicit multi-scale feature pyramids (e.g., PVTv2, Wang et al., 2022) may offer better native support for MTL by naturally producing features at multiple semantic levels. A paper that systematically compares different backbones under a controlled MTL protocol on 3+ tasks would tell us whether switching from MViTv2-S to, say, ConvNeXt-Base could reduce the MTL-to-ST gap by 5-10%. We already know ConvNeXt-MTL achieves 0.468 mAP on IndustReal detection (our reference), but we need the published comparison data.

**Question A4: How should feature routing work — which backbone layers should feed which task heads to minimize negative transfer?**

Currently, all four heads draw from the final backbone layer: detection uses P3/P4/P5 features (intermediate stages via hooks), activity/pose use the cls_token (final layer), and PSR uses P5 (final block before head). This creates a situation where detection alone has access to intermediate features, while activity and pose compete for the same cls_token representation. Cross-stitch networks (Misra et al., 2016), NDDR-CNN (Gao et al., 2019), and routing-by-uncertainty (Kendall et al., 2018) offer different philosophies for which layers feed which tasks. A paper that systematically evaluates routing strategies for 4 disparate tasks — particularly one where detection uses earlier features while semantic tasks use later features — would give us a principled basis for redesigning our feature routing. Our current ad-hoc routing (intermediate hooks for detection, final features for others) likely leaves significant headroom.

**Question A5: Can separate detection necks (FPN vs. BiFPP vs. NAS-FPN vs. PANet) significantly close the MTL-to-ST detection gap when operating on shared backbone features?**

We use a custom LightweightBiFPN with EfficientDet-style weighted fusion and 3D convolutions. The choice of detection neck determines how effectively the shared backbone's multi-scale features are fused for object detection. BiFPN (Tan et al., 2020) was designed for EfficientDet's single-task setup; NAS-FPN (Ghiasi et al., 2019) was discovered via architecture search for COCO detection; PANet (Liu et al., 2018) adds a bottom-up path. A paper that compares these necks in a controlled MTL setting (same backbone, same detection head, same other tasks) would isolate whether the neck architecture is a bottleneck for our 0.25-0.45 mAP@0.5 detection range. We currently observe detection mAP around 0.25-0.45 from MTL but 0.40-0.55 from ST, and we need to know whether a better neck accounts for 10% or 50% of that gap.

**Question A6: What architectures from multi-modal learning (cross-attention, modality-specific encoders, fusion gating) could transfer to multi-task learning with disparate output types?**

Multi-modal learning routinely handles inputs of fundamentally different types (text + image, audio + video, etc.), which parallels our challenge of handling outputs of different types (boxes, class logits, temporal sequences, 6D vectors). Methods like cross-modal attention (Tsai et al., 2019), modality-specific encoders with late fusion (Baltruvsaitis et al., 2018), or Mixture of Modality Experts (Wang et al., 2020) could be adapted to multi-task learning by treating each task as a "modality" with its own representation needs. A paper that formalizes multi-task learning as a multi-modal fusion problem and demonstrates positive transfer on 4+ tasks would provide a theoretical framework for our architecture design. This is a speculative but potentially high-impact direction given the maturity of multi-modal fusion literature.

**Question A7: Are there published MTL architectures where the combined model achieves HIGHER accuracy on some tasks than any single-task specialist (i.e., positive transfer exceeding ST)?**

This is the holy grail for our paper's narrative. If any published MTL system beats its single-task baselines on all tasks, the mechanism that enables this would directly inform our architecture design. Known candidates include cross-task feature sharing where detection features improve activity (e.g., object presence informs action), or where pose features improve PSR (e.g., head direction indicates attention, which correlates with assembly state). Cross-stitch networks (Misra et al., 2016) showed modest positive transfer on NYUv2 tasks, and UberNet (Kokkinos, 2017) showed that joint training across 7 tasks improved some tasks. A paper documenting the conditions under which MTL exceeds ST — particularly for a task set including detection — would justify architectural investment in cross-task feature exchange mechanisms.

**Question A8: How should the trade-off between shared and task-specific parameters be allocated across the network depth for 4+ disparate tasks?**

Our model has ~34.5M shared backbone parameters and ~10M task-specific head parameters — roughly 77% shared, 23% task-specific. This ratio is arbitrary, set by the choice of MViTv2-S as backbone with lightweight heads. Papers on branched networks (e.g., BERT-MTL, Clark et al., 2019; or the branched architecture in Vandenhende et al., 2021) suggest that allocating more task-specific capacity at higher layers can reduce negative transfer. A paper that systematically varies the depth at which task-specific branching occurs across 4+ diverse tasks would tell us whether our 77% shared ratio is suboptimal. If the literature suggests 50-60% shared is optimal for 4+ diverse tasks, we would need to introduce task-specific adapters or parallel branches at intermediate backbone stages.

## Section B: MTL Optimization & Loss (7 questions)

---

**Question B1: What loss weighting schemes have been empirically proven to outperform Kendall uncertainty weighting (with capped precisions) on 4+ task MTL benchmarks?**

We use Kendall uncertainty weighting (Kendall et al., 2018) with capped precision log-variances: det <= 1.5, act <= 1.0, psr <= 0.5, pose <= 2.0 (in log space). Despite being one of the most cited MTL weighting methods, Kendall weighting assumes homoscedastic uncertainty that is task-specific and input-independent, which may be a poor fit for tasks with heteroscedastic noise (like detection, where uncertainty varies with object size). Alternative methods like Dynamic Weight Average (DWA; Liu et al., 2019), GradNorm (Chen et al., 2018), or Impartial MTL (IMTL; Liu et al., 2021) could offer more stable weighting. A paper that compares 5+ weighting methods on a 4+ task benchmark (ideally including detection + classification + regression) would tell us whether switching from Kendall weighting could gain 5-15% on our worst-performing heads. We already know Kendall weighting can collapse when one task's loss dominates (the "Kendall collapse" we document in doc 208), and we need to know which method is most robust to this failure mode.

**Question B2: How can gradient conflicts be resolved more effectively than PCGrad, and what published results exist for CAGrad, GradDrop, RotoGrad, Nash-MTL, or other alternatives on 4+ task MTL?**

We use PCGrad (Yu et al., 2020) with per-step random task ordering. PCGrad projects conflicting gradient components onto the normal plane of the conflicting gradient, which is a simple and elegant mechanism but has known limitations: it treats all conflicts equally regardless of magnitude, it doesn't consider higher-order interactions among 3+ tasks, and it can still converge to Pareto-suboptimal solutions. CAGrad (Liu et al., 2021) explicitly optimizes for a minimum desired improvement across all tasks; GradDrop (Chen et al., 2020) randomly drops gradient elements based on sign agreement; RotoGrad (Javaloy et al., 2022) rotates task gradients to reduce conflict; Nash-MTL (Navon et al., 2022) frames MTL as a Nash bargaining problem. A paper that compares these methods on 4+ tasks including detection would tell us which gradient surgery technique best handles the type of conflicts present in our setup (detection vs. pose, activity vs. PSR). We need published evidence on which method converges fastest and achieves highest Pareto front quality.

**Question B3: What published results exist for MTL optimizers (beyond gradient surgery) on benchmarks with 4+ tasks, and what Pareto front improvements do they demonstrate?**

Beyond loss weighting and gradient surgery, several papers propose end-to-end MTL optimization frameworks: MGDA (Desideri, 2012; Sener & Koltun, 2018 for multiple-gradient descent), FAMO (Liu et al., 2023), and InvPT (Ye et al., 2023). MGDA finds a common descent direction that improves all tasks simultaneously; FAMO uses a fairness-based adaptive weighting; InvPT models task interactions via pairwise influence functions. A comprehensive survey or benchmark paper that reports Pareto front quality (hypervolume, distance to reference point) for 6+ MTL optimizers on a 4+-task benchmark would let us situate our current PCGrad + Kendall approach within the optimization landscape. We suspect our method may be leaving 10-20% Pareto front headroom, and we need quantified evidence to justify the engineering investment in switching.

**Question B4: Can task prioritization, curriculum learning, or staged training strategies improve MTL convergence as measured by published results on 4+ task benchmarks?**

We currently train all tasks jointly from epoch 1 with a Kendall act_ramp that linearly increases the activity weight over the first 5 epochs. More sophisticated strategies include: (1) progressive training where easy tasks are introduced first and harder tasks later (Bengio et al., 2009), (2) self-paced learning where the model chooses which tasks to focus on based on current loss (Kumar et al., 2010), (3) task-specific learning rate schedules, or (4) alternating task training rather than joint training. A paper demonstrating that staged or curriculum-based training significantly outperforms joint training on 4+ tasks (especially where one task dominates like pose in our setup) would provide a practical, low-engineering-cost improvement path. We already know from our overfit probes that pose converges fastest (57 steps), while activity and detection show no learning in the probe condition, suggesting a staged training order (pose first as warmup, then detection, then activity/PSR) might produce better outcomes.

**Question B5: What published MTL results claim beating single-task baselines on ALL constituent tasks, and what optimization methods do they use?**

This is the strongest claim an MTL paper can make, and identifying the common methodological thread across papers that achieve it would be extremely valuable. Candidates include: Yu et al. (2020) showing PCGrad improves all tasks on NYUv2 and Cityscapes; Liu et al. (2021) showing CAGrad matches or exceeds ST on NYUv2; Navon et al. (2022) showing Nash-MTL improves all tasks on some benchmarks. If we can identify a consistent pattern — e.g., all papers that beat ST use some form of gradient deconfliction plus task-specific normalization — we would have strong evidence for what to prioritize in our optimization pipeline. We specifically need to know whether any published paper beats ST on a task set that includes object detection with small objects, as this is our hardest case.

**Question B6: Can meta-learning or learned loss weighting schemes (e.g., MetaBalance, RL-weighted losses) outperform hand-tuned or uncertainty-based schedules on 4+ task MTL?**

Our Kendall weighting uses learned log-variances per task, making it a form of learned loss weighting. However, more sophisticated meta-learning approaches could learn: (1) task weights as a function of training epoch or validation performance, (2) per-parameter task importance via meta-gradients, or (3) gating networks that select which task's gradient to follow at each step. MetaBalance (He et al., 2022) and similar methods treat the weighting as a bi-level optimization problem. A paper that demonstrates meta-learned weighting strategies significantly outperforming Kendall weighting on 4+ tasks would justify the additional complexity. We need to know whether the meta-learning overhead pays off in terms of final task accuracy, or whether simpler methods (like fixing weights after the first 10 epochs) perform comparably.

**Question B7: What is the state of the art in MTL optimization as of 2025-2026, including the latest advances beyond PCGrad, CAGrad, and Nash-MTL?**

The MTL optimization field is moving rapidly, and methods published in 2024-2026 may offer substantial improvements. Advances could include: (1) transformer-based gradient aggregation that learns task interactions, (2) diffusion-based multi-task optimization, (3) large language model guided task weighting, or (4) methods leveraging the Pareto front approximation for dynamic trade-off selection. A survey or state-of-the-art paper from 2025-2026 that benchmarks the latest methods on standard MTL benchmarks (Taskonomy, NYUv4, PASCAL-Context) would tell us whether we are using outdated optimization methods. Our current stack (Kendall 2018 + PCGrad 2020) is 6-8 years old, and we need to know the magnitude of improvement offered by 2024+ methods — is it 5% relative improvement or 20%?

## Section C: Detection in MTL (6 questions)

---

**Question C1: What published evidence explains why detection suffers most from feature sharing in MTL, and what architectural or optimization interventions specifically address this?**

Across our experiments, detection shows the largest MTL-to-ST gap (MTL/ST ratio <= 0.60 based on doc 208 expectations). This is consistent with broader MTL literature: dense prediction tasks (detection, segmentation) typically degrade more under feature sharing than global tasks (classification, regression). The hypothesized reasons include: (1) detection requires precise spatial information that is lost in globally pooled features, (2) detection gradients from background anchors (vast majority of anchors are negative) dominate the gradient signal and drown out other tasks, (3) the FPN multi-scale features compete for backbone capacity with semantic tasks. A paper that systematically diagnoses detection degradation in MTL and proposes a targeted fix (e.g., task-specific attention layers, gradient clipping for detection, or separate normalization statistics) would be directly applicable to our setup. We need to know whether the primary cause is architectural (feature resolution), optimization (gradient conflict), or data (anchor imbalance), so we can prioritize the right fix.

**Question C2: What detection-specific architectures (BiFPN, NAS-FPN, YOLO-family decoupled heads, RT-DETR) have been shown to work in MTL settings, and what detection mAP do they retain relative to ST?**

Our detection head uses a decoupled classification + regression design inspired by YOLOX, operating on a BiFPN neck with weighted fusion. While decoupled heads and FPN variants were designed for single-task detection, their MTL-specific behavior is not well documented. A paper that evaluates different detection heads and necks in a controlled MTL setting (same backbone, same auxiliary tasks) would tell us whether the detection architecture itself needs to change for MTL, or whether the current architecture is adequate and the bottleneck is elsewhere. We already know our detection head achieves 0.40-0.55 mAP@0.5 in ST but only 0.25-0.45 in MTL, and we need to decompose that gap into architecture-specific vs. optimization-specific components.

**Question C3: How to handle small objects (~20px at 224px input resolution) in a shared backbone MTL setup, particularly regarding anchor assignment and feature resolution?**

Our assembly parts are approximately 20 pixels at 224px input resolution, which means they span roughly 3 cells on the P3 feature grid (28x28). This is at the boundary of what standard object detectors can handle, and the problem is exacerbated in MTL because the shared backbone cannot dedicate disproportionate capacity to fine-grained spatial features. Techniques like feature pyramid refinement (Dilated FPN), high-resolution detection heads operating on P2 features (which we currently skip due to semantic concerns), or learned anchor assignment (ATSS, Zhang et al., 2020; SimOTA, Ge et al., 2021) could help. A paper that specifically addresses small object detection under MTL conditions (shared backbone with 3+ other tasks) would provide a template for our detection pipeline. We need to know whether the small-object problem is fundamentally unsolvable at 224px input or whether architectural innovations can bridge the gap.

**Question C4: Can detection benefit from cross-task feature exchange with activity, PSR, or pose heads — for example, using activity class context to prime detection or using pose attention to focus detection on relevant spatial regions?**

This is one of the most promising hypotheses for positive transfer: if the activity head recognizes "screwdriving" and the pose head knows the operator is looking at a specific region, the detection head could use this contextual information to focus on task-relevant objects. Mechanism could include: (1) injecting activity logits as contextual priors into the detection classification head, (2) using pose attention maps to gate detection features spatially, (3) using PSR temporal state to filter implausible object detections. A paper demonstrating cross-task feature sharing between a detection task and semantic/pose tasks would provide a blueprint for our architecture. This is particularly important because cross-task sharing could convert negative transfer into positive transfer, transforming our MTL story from "detection degrades" to "detection improves from context."

**Question C5: What published MTL papers report competitive detection mAP against single-task baselines (mAP retention > 85%), and what specific techniques do they use?**

While many MTL papers include semantic segmentation as their dense prediction task, relatively few include full object detection (classification + bounding box regression). Papers that do report competitive detection retention are rare and valuable. Examples include YOLACT (Bolya et al., 2019) for instance segmentation, Detectron2 MTL variants, and some multi-task 3D detection papers. A comprehensive review of MTL papers that include standard COCO-style detection (with mAP@0.5:0.95) and report MTL/ST ratios would tell us whether 85%+ retention is achievable for detection or whether our 60% target is realistic. If the literature consistently shows detection retention < 70% in MTL, we can set appropriate expectations and frame our contributions around other metrics (efficiency, cross-task synergy).

**Question C6: How should anchor assignment strategies (TAL, ATSS, SimOTA, YOLOv9's TaskAlignedAssigner) differ in MTL vs. single-task detection settings?**

Our detection pipeline uses a fixed IoU threshold (0.5 positive, 0.4 negative) with top-k force matching (k=9, floor=0.2). In single-task detection, anchor assignment strategies have evolved from fixed IoU thresholds (RetinaNet) to dynamic assignment (ATSS adapts thresholds based on statistics; SimOTA assigns based on cost; TAL uses alignment matching). These strategies may behave differently under MTL because the shared backbone produces features that are not optimized purely for detection. A paper that studies how dynamic anchor assignment performs under MTL gradient interference would tell us whether adapting our assignment strategy could reclaim detection performance. We particularly need to know whether the top-k force match strategy works well when detection gradients are competing with other tasks' gradients.

## Section D: Activity Recognition in MTL (6 questions)

---

**Question D1: How to handle extreme long-tail class distributions (75 classes across assembly steps, power-law distribution) in a shared-backbone MTL setting?**

Our activity recognition has 75 classes with a power-law distribution where tail classes may have 1-5 samples. In single-task settings, logit adjustment (Menon et al., 2020), balanced sampling, and class-aware margin losses (ArcFace, Deng et al., 2019) are standard. However, in MTL, the long-tail problem interacts with shared representations: the backbone may learn features that are dominated by head classes, and task-specific logit adjustment may be insufficient to recover tail class accuracy. We currently use logit adjustment with tau=1.0, three-layer MLP (768->2048->1024->75), and balanced sampling. A paper that addresses long-tail activity recognition specifically in MTL settings (where the backbone is shared with detection and regression tasks) would tell us whether specialized loss functions or architectural modifications are needed beyond single-task long-tail methods. We need to know if our 30-45% realistic top-1 accuracy is constrained by the long tail, by feature sharing, or by the inherent difficulty of 75 fine-grained assembly actions.

**Question D2: What temporal modeling approaches (attention pooling, transformer encoder, LSTM, TSM, TSN, SlowFast branches) outperform simple cls_token pooling for activity recognition in an MTL context?**

We currently feed the cls_token (global pooled representation from MViTv2-S) through a 3-layer MLP for 75-class activity classification. This discards all spatial and temporal structure in the feature map, relying entirely on the transformer's learned global representation. More sophisticated temporal modeling approaches like TSM (Lin et al., 2019), temporal attention pooling, or a lightweight transformer on spatial-temporally pooled features could capture finer-grained motion patterns. A paper that compares temporal modeling approaches for activity recognition in a shared-backbone MTL setting (where additional computation must be balanced against other tasks' needs) would help us decide whether upgrading the activity head from our 3-layer MLP to a temporal model would close the gap to the 65.25% SOTA. We need to know the cost-benefit trade-off: does a temporal activity head add 5M parameters for 5% gain, or can a lightweight attention pool achieve similar gains with negligible overhead?

**Question D3: Can verb-noun factorization or hierarchical classification (e.g., coarse-to-fine, part-based, or compositional action models) improve MTL activity accuracy compared to flat 75-class classification?**

Our 75 activity classes likely exhibit compositional structure: they combine actions (screw, insert, place) with objects (screw, bracket, panel). A factorization approach that predicts verb + noun separately and combines them could: (1) reduce the effective class count per head, (2) enable sharing of verb representations across objects and vice versa, (3) handle novel verb-object combinations at test time. In an MTL context, verb representations could be shared with the PSR head (which predicts procedural states) and noun representations with the detection head (which identifies objects). A paper demonstrating verb-noun factorization in an MTL setting with detection and temporal tasks would provide strong architectural guidance. We marked factorization as "next-paper territory" in doc 208, but if published evidence shows 10-20% gains in MTL settings, it could justify the label-space redesign.

**Question D4: What published results show MTL activity recognition accuracy within 5% of the single-task baseline, and what conditions enable this (architecture, loss weighting, data augmentation)?**

This is a critical question for setting our paper's contribution in context. If published evidence consistently shows that MTL activity recognition retains 95%+ of ST accuracy when certain conditions are met (e.g., sufficient backbone capacity, proper gradient handling, balanced task weighting), we can benchmark our current 70% retention target against state-of-the-art expectations. Conversely, if the literature shows that activity recognition typically degrades 15-30% in MTL, our 70% target may be ambitious and our 30-45% realistic range may need reframing. We need a systematic review of MTL activity recognition papers that report both MTL and ST numbers, ideally on fine-grained (>50 class) datasets similar to our assembly actions.

**Question D5: How can the activity head benefit from features or logits produced by detection, PSR, or pose heads in a principled cross-task sharing architecture?**

This is a cross-task positive transfer hypothesis: the detection head identifies objects (screwdriver, bracket), the PSR head identifies procedural state (step 3 of 11), and the pose head identifies operator gaze direction. Each of these could provide informative context for activity recognition. For example, knowing that a screwdriver is detected in the frame and that the procedure is in step 3 strongly constrains the possible activity classes (it must be a screwdriving action, not a placement action). A paper that implements cross-task feature sharing in MTL where semantic heads inform classification heads via attention or conditional computation would provide an architectural blueprint. We need to know whether existing cross-task fusion methods (e.g., cross attention between task representations, or learned task affinity matrices) are mature enough for our 4-task setup.

**Question D6: What data augmentation strategies specifically improve activity recognition accuracy in an MTL context where augmentations must be compatible with all tasks?**

Our training pipeline applies spatial augmentations (flip, crop) and temporal stride (stride 3 = 1.6s windows). More aggressive video augmentations like MixUp (Zhang et al., 2018), CutMix (Yun et al., 2019), or video-specific augmentations (VMix, Yun et al., 2020; VideoRandAugment) could improve activity generalization. However, in MTL, augmentations must be compatible with all tasks — for example, MixUp would corrupt bounding box labels for detection, and temporal cropping must preserve PSR transition events. A paper that studies augmentation strategies specifically designed for MTL video understanding (where different tasks have different augmentation sensitivities) would help us expand our augmentation pipeline without breaking task compatibility. We need to know whether aggressive augmentations that help activity (like temporal warping) can be applied without degrading detection mAP.

## Section E: PSR / Temporal State Detection (6 questions)

---

**Question E1: How to handle extreme class imbalance (<1% positive frames, 11-component binary labels) in MTL temporal prediction, specifically for procedural assembly state detection?**

Our PSR task has 11 binary components (assembly sub-steps), each with <1% positive frames, meaning the model sees over 100 negative frames for every positive transition. In MTL, this imbalance interacts with other tasks' gradients: the PSR loss (binary focal loss, alpha=0.25, gamma=2) may be numerically dominated by the activity CE loss (loss ~5) or detection focal loss (loss ~0.5). A paper that specifically addresses extreme temporal imbalance in MTL settings — perhaps through gradient normalization per task, focal loss with task-specific gamma, or balanced sampling across time and tasks — would directly inform our PSR head configuration. We already know that our PSR overfit probe succeeded (loss drove near zero, 91% positive predictions), but the real MTL setting introduces competition for backbone features that may prevent PSR from reaching the low-loss regime.

**Question E2: What architectures bridge the gap between per-frame prediction (our current approach) and event-level metrics (transition F1 with ±3 frame tolerance)?**

We currently predict per-frame binary logits (11 components x 8 frames = 88 outputs per clip) and compute event-F1@±3 by detecting transitions and allowing 3-frame tolerance. This two-stage approach (per-frame prediction -> event detection -> F1) creates a disconnect between the loss function (per-frame binary focal loss) and the evaluation metric (event-level F1). A paper that proposes transition-aware architectures — such as predicting transition probabilities directly, using CRF or HMM layers for temporal consistency, or employing set-prediction for events — could close this gap. In MTL, the challenge is that such architectures may need more parameters or specialized layers that compete with other tasks for backbone capacity. We need to know whether published transition-specific architectures achieve event-F1 > 0.5 in MTL or whether the per-frame approach is fundamentally limited to the 0.10-0.35 range we project.

**Question E3: How can procedural or assembly prior knowledge — such as the known linear ordering of assembly steps or the monotonic nature of state transitions — be incorporated into a learned MTL model as architectural inductive bias?**

Assembly procedures follow a known linear sequence: step N must complete before step N+1 can begin. This monotonicity constraint (once-on, stays-on) is a powerful prior that we currently apply only at evaluation time via post-processing (median filter, per-recording decoding). A paper that embeds monotonicity or temporal ordering constraints directly into the loss function (e.g., via ordering penalties or constrained optimization), the architecture (e.g., via causal masking beyond the transformer's built-in causality, or via cumulative sum layers), or the training procedure (e.g., via curriculum learning over the step sequence) could significantly improve PSR accuracy without requiring more data. In MTL, such inductive biases are particularly valuable because they reduce the data required to learn the temporal structure, freeing backbone capacity for other tasks. We need to know whether published procedural understanding methods (e.g., from assembly or recipe understanding literature) have been integrated into end-to-end learned models.

**Question E4: What published work uses monotonicity or temporal ordering constraints as architectural inductive bias in neural networks for state detection or procedural understanding?**

Beyond post-processing, monotonicity can be embedded architecturally via: (1) monotonic neural networks (Sill, 1998; Daniels & Velikova, 2010) that guarantee non-decreasing outputs, (2) cumulative sum or integral layers that accumulate evidence over time, (3) hidden Markov models with monotonic transition matrices, or (4) ordinal regression losses for ordered states. A paper that successfully integrates any of these into a learned state detection model — particularly in an MTL or video understanding context — would provide a concrete implementation path. We currently gain +0.05-0.15 event-F1 from post-processing monotonicity, and we hypothesize that architectural enforcement could double this gain. We need published evidence on the magnitude of improvement from architectural vs. post-processing monotonicity.

**Question E5: Can transformer or attention-based temporal models (with causal masking, cross-attention to detection features, or temporal convolutional patterns) significantly outperform our simple 2-layer causal transformer for state detection in MTL?**

Our PSR head uses a 2-layer causal transformer (d=256, nhead=4, ff=1024) on spatially-pooled P5 features. While this is a lightweight and sensible architecture, more sophisticated temporal models could include: (1) longer-range temporal transformers that capture dependencies across the full assembly recording rather than the 8-frame window, (2) cross-attention layers that attend to detection features (objects present in the frame should inform state predictions), or (3) temporal convolutional networks (TCN, Bai et al., 2018) that may better capture sharp transition boundaries than transformers. A paper that systematically compares temporal architectures for state detection in a shared-backbone setting would guide our head redesign. We need to know whether the 2-layer transformer's limitation is capacity (need more layers) or receptive field (need longer context).

**Question E6: How to handle the windowed prediction problem for PSR — where T=8 frame predictions must map to event-level metrics that span entire recordings with varying numbers of transitions?**

Our model processes 8-frame windows (T=8 after backbone pooling, from T=16 input with stride 3) but event-F1 is evaluated over entire recordings with 2-5 transitions per 11-step procedure. This mismatch means each window typically contains at most 1 transition, and most windows contain none. The model must therefore learn to detect transitions from very local evidence, with no access to the longer procedural context. A paper that addresses the window-to-recording aggregation problem — through recurrent state propagation, memory augmentation, or post-processing that enforces procedural constraints across windows — could improve event-F1 without changing the per-window architecture. In MTL, this is a particularly tractable improvement because it operates in post-processing or through a lightweight memory module that wouldn't compete with other tasks for backbone capacity.

## Section F: Efficiency & Benchmarking (5 questions)

---

**Question F1: What is the established methodology in the MTL literature for computing parameter/FLOPs efficiency ratios (MTL vs. single-task ensemble), and what ratios are considered "publishable" at venues like WACV, CVPR, or ICCV?**

Our efficiency claim is ~2.2x parameter efficiency (~45M vs ~100M for four specialists). However, methodologies for computing this ratio vary widely: some papers compare against individual task experts for each task, others compare against a single multi-task model of equal total capacity, and others use normalized FLOPs accounting for the shared forward pass. A survey or reference paper that defines standard efficiency metrics for MTL — including how to compute FLOPs when the backbone is shared, how to account for task-specific processing, and what efficiency ratios are competitive at CVPR/WACV — would give us a defensible methodology for our paper. We need to know whether our 2.2x claim is strong enough for publication or whether we need to redesign for higher efficiency (targeting 3x+) to meet conference standards.

**Question F2: How should statistical significance (confidence intervals, paired bootstrap, or hypothesis tests) be computed for MTL vs. ST comparisons across multiple tasks with different metrics (mAP, top-1 accuracy, MAE, event-F1)?**

Comparing MTL to ST performance across 4 tasks with different metric types (detection: mAP@0.5; activity: top-1 accuracy; pose: MAE in degrees; PSR: event-F1@±3) presents a statistical challenge. Standard methods for multi-task significance testing (e.g., Bonferroni-corrected paired t-tests, or corrected resampled t-tests) may not apply cleanly because the same test set is used for all comparisons and the metrics are not directly commensurable. A paper or established methodology from the MTL community that defines how to report significance for multi-task results — including how to handle multiple metrics, correlated test sets, and the challenge of comparing against multiple ST baselines — would be essential for our paper's rigor. We need to know the accepted practice at WACV/CVPR for reporting confidence intervals in MTL.

**Question F3: What MTL benchmarks (Taskonomy, NYUv2, Cityscapes, PASCAL-Context, CelebA) have the most published MTL papers, and what are the typical MTL/ST retention ratios reported on each benchmark?**

Understanding the MTL retention ratios achievable on established benchmarks would let us contextualize our own results. For example, if NYUv2 papers consistently report 90%+ retention for 3-task MTL (segmentation, depth, normal), but Cityscapes detection papers report 70-80% retention, we would have evidence that detection is inherently harder to retain in MTL. A meta-analysis or survey of MTL papers across the major benchmarks, reporting the distribution of MTL/ST retention ratios per task type, would provide the empirical grounding for our paper's expectations. We need to know whether our projected MTL/ST ratios (pose: 95%, activity: 70%, detection: 60%, PSR: unknown) are in line with published norms or represent unusual difficulty.

**Question F4: How to fairly report FLOPs for MTL when the backbone computation is shared across tasks — should we report total inference FLOPs for one forward pass, or per-task amortized FLOPs?**

The common practice is to report total FLOPs for the MTL model performing all tasks simultaneously, compared to the sum of FLOPs for each ST model performing its task. However, this creates a subtle unfairness: if the MTL model always computes all task outputs, but a practitioner may only need some tasks at inference time, the comparison overestimates efficiency. Conversely, if the ST models share computational patterns (e.g., both use the same backbone architecture), the ST comparison may underestimate their true cost (since separate models means separate memory, model loading, etc.). A paper or analysis that defines best practices for FLOPs reporting in MTL — including scenarios with partial task execution, variable input resolution, and hardware-specific latency — would inform how we present our efficiency claims. We need to know whether latency (FPS) or FLOPs is the more persuasive metric for our specific use case (real-time assembly monitoring).

**Question F5: What efficiency metrics (latency, throughput, peak memory, model storage) matter most for real-world deployment of MTL models, and what published benchmarks report these for video understanding MTL?**

While parameter count and FLOPs are standard in academic papers, real-world deployment cares about: (1) latency on target hardware (e.g., Jetson Orin for assembly line monitoring), (2) throughput in frames per second, (3) GPU memory for batch processing, and (4) storage and loading time for model weights. A paper that benchmarks MTL video understanding models on deployment-relevant hardware (edge devices, server GPUs) with these metrics would tell us which of our architectural choices have the most impact on deployability. We need to know, for example, whether our BiFPN neck's 3D convolutions are a latency bottleneck, or whether the MViTv2-S backbone's memory footprint is the primary constraint.

## Section G: Training & Data Strategy (6 questions)

---

**Question G1: What training recipes (batch size, learning rate schedule, warmup, optimizer hyperparameters) are proven for MTL with transformer-based video backbones and 4+ disparate tasks?**

Our training recipe: batch size 16 (effective, with gradient accumulation), AdamW optimizer (lr=1e-4, weight decay=0.05), cosine decay schedule with linear warmup, 50 epochs (~39k batches per epoch). This recipe is inherited from single-task video classification and may be suboptimal for MTL. MTL-specific training considerations include: (1) whether lower learning rates help when gradients are aggregated across tasks, (2) whether longer warmup helps the shared backbone accommodate multiple tasks, (3) whether different tasks benefit from different learning rates (detection may need higher lr than pose). A paper that systematically studies MTL training recipes for multi-task video transformers would provide principled guidance. We need to know whether switching from AdamW to SGD with momentum, or from cosine decay to step decay, could improve convergence across all four tasks.

**Question G2: How can data sampling balance across tasks with different annotation densities (all frames have pose labels, only some frames have detection boxes, activity labels are per-clip, PSR labels are per-frame)?**

Our four tasks have fundamentally different annotation densities: head pose is annotated on every frame, detection boxes are available on a subset of frames, activity labels are per-clip (one label per 16-frame window), and PSR transition labels are per-frame but sparse (<1% positive). Standard uniform sampling would over-represent densely annotated tasks (pose) and under-represent sparsely annotated tasks (PSR, detection). A paper that addresses multi-task data sampling strategies — including task-specific sampling probabilities, mixed mini-batch construction, and handling of missing annotations for some tasks in each sample — would help us design a training data pipeline that gives each task sufficient exposure. We currently use a single dataset that contains all annotations for each clip, but the imbalance in annotation density may still bias training toward tasks with more non-zero labels.

**Question G3: Can synthetic data, self-supervised pretraining, or large-scale image/video pretraining (beyond Kinetics-400) close the data scarcity gap for our assembly domain, particularly for MTL where multiple tasks compete for limited annotations?**

Our Kinetics-400 pretrained MViTv2-S backbone provides generic video features, but the assembly domain has domain-specific visual patterns (small metal parts, hands with tools, assembly fixtures) that may not be well represented in Kinetics. Additional pretraining strategies could include: (1) self-supervised video pretraining (VideoMAE, Tong et al., 2022; MaskFeat, Wei et al., 2022) on in-domain assembly videos, (2) synthetic data generation (rendering assembly scenes with known ground truth for all four tasks), or (3) large-scale image-text pretraining (CLIP, Radford et al., 2021; DINOv2, Oquab et al., 2023) adapted to video via temporal aggregation. A paper that demonstrates MTL improvement from domain-specific pretraining or synthetic data augmentation — particularly for industrial/manufacturing domains — would justify the data generation investment. We need to know whether self-supervised pretraining on our assembly videos (unlabeled) could close 10-20% of the MTL gap without requiring additional annotations.

**Question G4: What curriculum learning or staged training strategies have been published for MTL, and what performance improvements do they demonstrate on 4+ task benchmarks?**

We implement a simple curriculum by ramping the activity weight linearly over the first 5 epochs via act_ramp. More sophisticated curricula could include: (1) "easy task first" — train pose and detection first (simpler losses), then add activity and PSR, (2) "head start" — pretrain each head independently for a few epochs before enabling full MTL, (3) "alternating freeze" — periodically freeze the backbone and train only heads, then freeze heads and train backbone. A paper that systematically compares curriculum strategies for MTL convergence would help us design a training pipeline that avoids the initial task competition that likely degrades our current results. We already implement head warm-starting (init heads from ST checkpoints), which is a form of curriculum, and we need to know whether staged training (sequential task introduction) offers additional benefits.

**Question G5: How many epochs are typically needed for MTL convergence vs. single-task convergence, based on published evidence across different benchmarks and task counts?**

We train for 50 epochs in both MTL and ST settings, based on the assumption that MTL and ST require similar training budgets. However, MTL may require more epochs because: (1) the shared backbone must learn features that serve multiple tasks, which may require more gradient updates to converge, (2) gradient conflicts may slow per-task convergence, (3) task weighting mechanisms (Kendall log-vars) need time to stabilize. A paper that documents the convergence behavior of MTL vs. ST across a range of datasets and task counts would tell us whether 50 epochs is adequate or whether we need 100+ epochs for MTL to match ST convergence. We already observe that some heads (pose) converge much faster than others (activity, detection), and we need to know whether extending training would allow activity and detection to catch up.

**Question G6: What early-stopping or checkpoint selection criteria work for multi-metric MTL models where the best checkpoint differs by task (pose peak at epoch 20, detection peak at epoch 45, activity still improving at epoch 50)?**

Our four tasks peak at different epochs: pose may plateau or overfit after epoch 20, detection may continue improving through epoch 50, activity may still be rising at epoch 50. This makes checkpoint selection non-trivial: the "best" model depends on the task weighting chosen for deployment. Strategies include: (1) validation-based selection using a composite metric (e.g., sum of normalized task scores), (2) Pareto front tracking across epochs to select the checkpoint with best trade-off, (3) SWA (stochastic weight averaging) over the last N checkpoints to average across convergence regimes. We currently use SWA with the last 5 periodic checkpoints, which is a pragmatic but unprincipled approach. A paper that defines checkpoint selection criteria for multi-metric MTL models — ideally with theoretical guarantees or empirical validation across diverse MTL benchmarks — would strengthen our evaluation methodology.

## Section H: Wildcard / Blue-Sky (6 questions)

---

**Question H1: Are there any published MTL models that beat SOTA on ALL their constituent tasks simultaneously (not just MTL/ST ratio > 1.0, but absolute SOTA)?**

This is the strongest possible MTL result. While many papers show MTL matching or slightly exceeding ST on some tasks, truly beating SOTA on all tasks is extremely rare. Candidates include some results in the Taskonomy paper (Zamir et al., 2018) where certain cross-task transfers improve accuracy, and some results in the cross-stitch networks paper (Misra et al., 2016). A systematic review of MTL papers that claim absolute SOTA on multiple tasks — with careful attention to whether the SOTA comparisons are fair (same input resolution, same backbone, same training data) — would set the upper bound on our ambitions. For our paper, if this result exists for a task set including detection, we would aim higher; if not, we can anchor our contribution around efficiency and first-of-its-kind claims rather than absolute SOTA.

**Question H2: What is the most surprising positive-transfer result in MTL literature — where two tasks that seem unrelated (e.g., detection and pose, or activity and depth) are shown to significantly benefit each other?**

The MTL literature contains several surprising positive transfer results: depth estimation helping semantic segmentation (both benefit from geometric understanding), or surface normal estimation helping object detection (both need fine-grained spatial features). Surprising positive transfers are valuable because they suggest architectural mechanisms for cross-task sharing that we might not expect but could exploit. A paper documenting an unexpected positive transfer between task types that matches our task set — for example, between head pose estimation and detection (pose indicates attention, which predicts object locations), or between activity recognition and PSR (actions and procedural states are temporally correlated) — would suggest a cross-task sharing mechanism we should implement. We already hypothesize that pose-to-detection transfer may exist but lack published evidence.

**Question H3: Can foundation models (CLIP, DINOv2, SAM, InternVideo, Video-LLaMA) be effectively adapted for multi-task learning with 4+ diverse task heads, or do their monolithic architectures resist efficient multi-task adaptation?**

Foundation models offer rich pretrained representations that could potentially serve as MTL backbones. CLIP (Radford et al., 2021) provides text-aligned visual features; SAM (Kirillov et al., 2023) provides segmentation-ready representations; DINOv2 (Oquab et al., 2023) provides self-supervised features that are surprisingly effective across many task types. However, these models were designed for single-task finetuning, and their architectural properties (e.g., CLIP's final pooling discards spatial information, SAM's ViT-H is 632M parameters) may make them suboptimal for MTL. A paper that systematically evaluates foundation model backbones for MTL — comparing their MTL/ST retention, Pareto front quality, and efficiency — would tell us whether to invest in adapting a foundation model as our backbone. We currently use MViTv2-S (designed for video classification, not MTL), and a foundation model may offer better starting representations at the cost of higher computational requirements.

**Question H4: What emerging architectures (Mamba, RWKV, Hyena, State Space Models in general) could revolutionize multi-task learning by providing more efficient and flexible sequence representations than transformers?**

State Space Models (SSMs) like Mamba (Gu & Dao, 2023), RWKV (Peng et al., 2023), and Hyena (Poli et al., 2023) offer linear-complexity sequence modeling that could replace transformers in MTL architectures. Their advantages for MTL include: (1) lower FLOP costs for long sequences (video features with large T), (2) more efficient cross-task attention mechanisms, (3) simpler gradient properties that may reduce task conflict. A paper that benchmarks SSM-based architectures for MTL video understanding — particularly with 4+ tasks and including dense prediction — would tell us whether investing in SSM-based backbones or heads could improve our efficiency and accuracy trade-offs. This is speculative but potentially high-impact given the rapid progress in SSM architectures.

**Question H5: Are there techniques from multi-agent RL (reward decomposition, credit assignment, agent coordination) or neural architecture search (NAS, evolutionary optimization of task-specific sub-networks) that are applicable to MTL architecture and optimization?**

MTL can be framed as a multi-agent problem where each task is an "agent" competing for shared computational resources (backbone capacity, gradient influence). Multi-agent RL techniques for credit assignment (e.g., COMA, Foerster et al., 2018; QMIX, Rashid et al., 2018) could be adapted to determine how much each task should influence the backbone update. NAS techniques for MTL (MTL-NAS, Gao et al., 2020) can learn task-specific sub-network architectures within a shared backbone. A paper that successfully applies multi-agent credit assignment or NAS to MTL architecture design — with demonstrable improvements over standard MTL methods on 4+ tasks — would provide a radically different approach to our problem. We need to know whether these techniques are mature enough for production use or remain academic curiosities.

**Question H6: If you could design the ideal MTL system from scratch with 2026 knowledge — given 4 diverse tasks (detection, activity, PSR, pose), limited annotations, and video input — what would the architecture, optimization, and training strategy look like?**

This is a synthesis question that would be answered not by a single paper but by a comprehensive survey or benchmark that integrates findings across multiple sub-fields. The ideal system might combine: (1) a foundation model backbone with task-specific adapters at multiple depths, (2) gradient deconfliction via Nash-MTL or a 2025 method, (3) cross-task attention layers that let detection features inform activity and vice versa, (4) joint training with curriculum learning and head warm-starting, (5) synthetic data augmentation for underrepresented tasks, (6) SWA or Pareto-optimal checkpoint selection, and (7) memory-efficient deployment via quantization or distillation. A paper or technical report that proposes an integrated MTL system design based on 2024-2026 literature and benchmarks it on a diverse 4+ task setup would be the closest thing to a "best practices" guide for our project. If no such synthesis exists, our own work could serve as a template for the 2025-2026 state of the art.
